# Reporters

Reference for Warden's output formats.

---

## 1. Reporter Modes

There are three reporters. Each one receives the same events and decides what to render.

| Mode | Purpose | Channel | Implementation |
|------|---------|---------|----------------|
| **TTY** | The interactive experience. Boxes, colors, spinners, inline code snippets. Designed to make findings easy to scan and act on. | stderr (progress), stdout (findings report) | `Reporter` class + Ink (`ink-runner.tsx`) |
| **Plain** | CI logs and piped output. Every line is timestamped and prefixed so you can grep, tail, and correlate with other CI output. Full findings report still goes to stdout. | stderr (timestamped progress), stdout (findings report) | `Reporter` (plain path), `renderSkillCI()` in `terminal.ts` |
| **JSONL** | Machine-readable run log. One JSON object per line for downstream tooling: dashboards, cost tracking, trend analysis, alerting. New logs are homogeneous chunk-result records, appended as each chunk finishes so `warden runs follow` and CI artifacts stay live. | File (always-on `.warden/logs/`, additional copy via `--output`) | `appendJsonlLine()` + `renderJsonlChunkLine()` in `jsonl.ts` |

**Plain format**: `[{ISO-8601}] warden: {message}` on stderr. Generated by `timestamp()` in `tty.ts`.

**GitHub Actions** runs in Plain mode. The Action workflow (`pr-workflow.ts`) adds GitHub-specific side effects on top: `::group::` blocks, `::warning::` annotations, `GITHUB_OUTPUT` variables, and GitHub API calls for PR reviews, check runs, and comment management. These are workflow orchestration, not a separate reporter. See [Section 8: Action Workflow Logging](#8-action-workflow-logging).

### Verbosity Levels

| Level | Flag | Effect |
|-------|------|--------|
| Quiet | `-q` | Final findings report only (stdout). No progress, no summary. |
| Normal | (default) | Skill/file/hunk progress + findings report + summary |
| Verbose | `-v` | + real-time per-finding lines during skill execution, per-hunk failure details, retry attempts |
| Debug | `-vv` | + token counts, prompt sizes, extraction methods, suggested fix descriptions, extraction failure output previews |

### Skill Execution

#### 5. `skill_start`

Fields: `name`, `displayName`, `fileCount`

**TTY:** Spinner icon + skill name + `[done/total files]` + finding count + running cost

**Plain:**
```
[2026-02-08T14:30:45.123Z] warden: Running security-review (3 file(s))...
```

**JSONL:** --

#### 6. `file_complete`

Fields: `filename`, `durationMs`, `findings[]`, `usage?`

**TTY:** File disappears from dynamic area (only running files shown). Checkmark + filename + severity dots + duration + cost appear in the post-unmount summary.

**Plain:**
```
[2026-02-08T14:30:46.500Z] warden:   security-review > src/api/auth.ts done 1.8s $0.01 1 finding(s)
```
Cost omitted when no usage data. Finding count omitted when zero.

**JSONL:** Captured by one `JsonlChunkRecord` per completed chunk

#### 7. `hunk_start`

Fields: `filename`, `hunkNum`, `totalHunks`, `lineRange`

**TTY:** -- (progress shown via file-level spinner)

**Plain:**
```
[2026-02-08T14:30:45.200Z] warden:   security-review > src/api/auth.ts [1/3] L10-45
```

**JSONL:** --

#### 8. `skill_complete`

Fields: `name`, `durationMs`, `findings` (count), `usage?`

**TTY:** Skill moves from spinner to checkmark in the dynamic area: `checkmark security-review [4.2s]` (green checkmark, duration dimmed). After all skills finish and Ink unmounts, the full per-skill + per-file breakdown prints to stderr.

**Plain:**
```
[2026-02-08T14:30:48.600Z] warden: security-review completed in 4.2s - 2 findings (1 high, 1 medium)
```

At `Verbose+`, per-finding lines follow:
```
[2026-02-08T14:30:48.601Z] warden:   [high] src/api/auth.ts:42: SQL injection risk
[2026-02-08T14:30:48.601Z] warden:   [medium] src/utils/helpers.ts:15: Unsafe type assertion
```

At `Debug`, suggested fix descriptions also shown.

**JSONL:** Derived from completed `JsonlChunkRecord` lines. New logs do not append a separate skill record.

#### 9. `skill_skipped`

Fields: `name`

**TTY:** `skip-icon {displayName} [skipped]`

**Plain:**
```
[2026-02-08T14:30:48.600Z] warden: security-review skipped
```

**JSONL:** --

#### 10. `skill_error`

Fields: `name`, `error`

**TTY:** `cross-icon {displayName} - {error}`

**Plain:**
```
[2026-02-08T14:30:48.600Z] warden: ERROR: security-review - API timeout after 30s
```

**JSONL:** A chunk record with `status: "error"`, empty `findings`, and an `error` object (`code`, `message`, `timestamp`) for chunk-level failures. Failures before any chunk runs may still use a legacy summary-only record.

### Findings

#### 11. `finding`

Fields: `id`, `severity`, `title`, `location?`, `description`, `suggestedFix?`, `confidence?`

Findings appear in two places: real-time during skill execution (stderr, verbosity-gated, Plain only) and in the final findings report (stdout, always shown).

**Real-time (stderr, `Verbose+`, Plain only):**

```
[2026-02-08T14:30:48.601Z] warden:   [high] src/api/auth.ts:42: SQL injection risk
```

TTY does not emit per-finding lines during execution. The Ink UI shows severity dots per-file in the progress display instead.

**Final report (stdout, always):** See "Findings Report" section below.

**JSONL:** Captured in the producing chunk's `findings[]` array

#### 12. `finding_summary`

Fields: `bySeverity` counts

**TTY:** Colored breakdown with severity dots

**Plain:** Embedded in `skill_complete` line as `{n} findings ({breakdown})`

**JSONL:** Derived by readers from chunk `findings[]`

### Findings Report (stdout)

After all skills complete, findings are rendered to stdout. This is separate from the progress events above (which go to stderr).

**TTY:** Box-rendered findings per skill via `renderSkillBoxTTY()`:
```
┌── security-review ────────────────────────────── [4.2s] ──┐
│ 2 findings: . 1 high  . 1 medium                          │
├───────────────────────────────────────────────────────────┤
│                                                           │
│ . (high) SQL injection risk in query builder              │
│   src/api/auth.ts:42  +0.8s                               │
│   42 | const query = buildQuery(userId);                   │
│                                                           │
│   User input is interpolated directly into a SQL query... │
│                                                           │
│   [high confidence]                                       │
│                                                           │
│   Suggested fix:                                          │
│   - const query = buildQuery(userId);                     │
│   + const query = buildQuery(sanitize(userId));           │
│                                                           │
├───────────────────────────────────────────────────────────┤
│                                                           │
│ . (medium) Unsafe type assertion on external input        │
│   src/utils/helpers.ts:15  +1.2s                          │
│   15 | const data = response as InternalType;              │
│                                                           │
│   External API response is cast without validation.       │
│                                                           │
│   [medium confidence]                                     │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

Each finding shows: severity badge, title, location with elapsed time, source code line (read from disk), description, confidence badge (when present), and suggested fix diff (colored: green for `+`, red for `-`, cyan for `@@`). The confidence badge appears after the description, separated by a blank line. It is colored by level: green for high, yellow for medium, red for low. When confidence is not provided, the badge and its blank line are omitted.

When fixable findings exist and interactive mode is active (TTY, no `--fix`, no `--json`, not quiet, not interrupted), suggested fix diffs are **suppressed** from the report because they will be shown in the interactive fix step-through (see Section below). All other finding fields (severity, title, location, description) still render normally.

**Plain:** Compact text per skill via `renderSkillCI()`:
```
security-review (4.2s) - 2 findings (1 high, 1 medium)
  WARN: 1 chunk failed to analyze
  Use -v for failure details
  [high] src/api/auth.ts:42 - SQL injection risk in query builder (+0.8s)
  confidence: high
  User input is interpolated directly into a SQL query string without parameterization.

  Suggested fix:
  - const query = buildQuery(userId);
  + const query = buildQuery(sanitize(userId));

  [medium] src/utils/helpers.ts:15 - Unsafe type assertion on external input (+1.2s)
  confidence: medium
  External API response is cast to an internal type without validation.
```

Per-skill warnings (`failedHunks`, `failedExtractions`) are shown inline, followed by a "Use -v for failure details" hint when either count is non-zero.

**JSONL:** Full structured records (see Section 3)

### Summary

#### 19. `analysis_complete`

Fields: `findingsCount`, `failedHunks?`, `failedExtractions?`, `skippedFiles?`, `usage?`, `totalDuration`, `traceId?`

This is a multi-line summary section, not a single event.

**TTY:**
```
SUMMARY
2 findings: . 1 high  . 1 medium
⚠  2 chunks failed to analyze
⚠  1 finding extraction failed
  Use -v for failure details
3 files skipped
Analysis completed in 4.2s · 5.0k in / 800 out · $0.01
Trace: abc123def456...
```
(Counts colored by severity. Warnings in yellow. Hint dimmed. Usage line dimmed. Trace ID dimmed, `Verbose+` only, only when Sentry is initialized.)

The `-v` hint appears only when there are failures (`failedHunks > 0` or `failedExtractions > 0`) **and** verbosity is below `Verbose`. At `Verbose+`, per-hunk details are already shown via events 16-18 so the hint is suppressed.

**Plain:**
```
[2026-02-08T14:30:56.000Z] warden: Summary: 2 findings (1 high, 1 medium)
[2026-02-08T14:30:56.000Z] warden: WARN: 2 chunks failed to analyze
[2026-02-08T14:30:56.000Z] warden: WARN: 1 finding extraction failed
[2026-02-08T14:30:56.000Z] warden: Use -v for failure details
[2026-02-08T14:30:56.000Z] warden: 3 files skipped
[2026-02-08T14:30:56.000Z] warden: Usage: 5.0k input, 800 output, $0.01
[2026-02-08T14:30:56.000Z] warden: Total time: 4.2s
[2026-02-08T14:30:56.000Z] warden: Trace: abc123def456...
```
Warnings and skipped files only shown when non-zero. The `-v` hint follows the same gating as TTY. Trace ID shown at `Verbose+` only, when Sentry is initialized.

**JSONL:** Derived by readers from chunk records. Run completion is tracked by the `.jsonl.done` sidecar marker.

### Interactive Fix Flow (TTY only)

After the summary, if fixable findings exist in TTY mode (no `--fix`, no `--json`, not quiet, not interrupted), the user steps through each finding one at a time.

Findings are displayed in reading order (file ascending, line ascending). Each finding shows:

```
. (high) [1/3] SQL injection risk in query builder
  src/api/auth.ts:42
  User input is interpolated directly into a SQL query...
  Use parameterized queries instead

  [high confidence]

  @@ -42,1 +42,1 @@
  -  const query = buildQuery(userId);
  +  const query = buildQuery(sanitize(userId));

[y]es / [n]o / [a]pply all / [s]kip all
```

Severity badge, counter, title, location, description, fix description (if present), confidence badge (when present), and colored diff.

**Prompt keys** (single keypress, no Enter):

| Key | Action |
|-----|--------|
| **y** | Accept this fix |
| **n** | Skip this fix |
| **a** | Accept this and all remaining fixes (stop prompting) |
| **s** | Skip this and all remaining fixes (stop prompting) |
| **q** | Same as **s** |

Any other key defaults to **n**.

After stepping through all findings, accepted fixes are applied in safe order (file ascending, line descending) to avoid line number shifts from earlier patches. Results are printed per fix:

```
✓ Applied: SQL injection risk in query builder
✓ Applied: Missing null check
✗ Failed: Unsafe cast: context mismatch at line 15
```

Then the fix summary renders (same as `--fix` mode):

```
FIXES
2 applied  1 skipped  1 failed
  ✗ Unsafe cast: context mismatch at line 15
```

**`--fix` flag:** Applies all fixes without prompting. No step-through.

**Plain/JSONL:** No interactive fix flow. The full findings report includes diffs as normal.

## 3. JSONL Specification

Formal schema: [`specs/jsonl-schema.json`](./jsonl-schema.json)

Example payloads: [`specs/jsonl-examples.jsonl`](./jsonl-examples.jsonl)

### Record Type

New run logs are homogeneous: every line is a chunk result. Each line is self-contained and can be parsed without reading a file header or trailer. Legacy skill, summary, and fix-evaluation records are still accepted by readers for old saved logs, but new normal run logs do not emit them.

### Chunk Record

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `schemaVersion` | `1` | yes | Record schema version |
| `run` | `RunMetadata` | yes | Run identity and repository metadata |
| `skill` | `string` | yes | Skill name |
| `model` | `string` | no | Model used for this chunk |
| `chunk` | `ChunkMetadata` | yes | File, chunk index, total chunks in file, and line range |
| `status` | `"ok" | "error" | "skipped"` | yes | Chunk outcome |
| `findings` | `Finding[]` | yes | Findings produced by this chunk, possibly empty |
| `usage` | `UsageStats` | no | Token usage and cost for this chunk |
| `durationMs` | `number` | yes | Chunk execution time |
| `auxiliaryUsage` | `Record<string, UsageStats>` | no | Auxiliary LLM usage for this chunk (extraction repair, etc.) |
| `error` | `SkillError` | no | Chunk failure details. Present when `status` is `"error"` |
| `skippedFiles` | `SkippedFile[]` | no | Files skipped before chunking. Present only on synthetic no-chunk records |

### Shared Types

**RunMetadata**: `{ timestamp: string, durationMs: number, cwd: string, runId: string, traceId?: string, model?: string, headSha?: string }`

**ChunkMetadata**: `{ file: string, index: int, total: int, lineRange: string }`

**UsageStats**: `{ inputTokens: int, outputTokens: int, cacheReadInputTokens?: int, cacheCreationInputTokens?: int, costUSD: number }` -- `inputTokens` is the total input token count; `cacheReadInputTokens` and `cacheCreationInputTokens` are subsets of it.

**Finding**: `{ id: string, severity: Severity, confidence?: Confidence, title: string, description: string, location?: Location, suggestedFix?: SuggestedFix, elapsedMs?: number }`

**Location**: `{ path: string, startLine: int, endLine?: int }`

**SuggestedFix**: `{ description: string, diff: string }`

**FileRecord**: Legacy reader type: `{ filename: string, findings: int, durationMs?: number, usage?: UsageStats }`

**SkippedFile**: `{ filename: string, reason: "pattern" | "builtin", pattern?: string }`

**SkillError**: `{ code: ErrorCode, message: string, timestamp?: string }`

**HunkFailure**: `{ type: "analysis" | "extraction", filename: string, lineRange: string, code: ErrorCode, message: string, preview?: string, attempts?: int }`

**ErrorCode**: one of `"auth_failed"`, `"sdk_error"`, `"subprocess_failure"`, `"max_turns"`, `"aborted"`, `"all_hunks_failed"`, `"skill_resolution_failed"`, `"extraction_invalid_json"`, `"extraction_unbalanced_json"`, `"extraction_no_findings_json"`, `"extraction_missing_findings_key"`, `"extraction_findings_not_array"`, `"extraction_llm_failed"`, `"extraction_llm_timeout"`, `"extraction_no_api_key"`, `"unknown"`. Stable public contract.

**BySeverity**: `{ high: int, medium: int, low: int }`. Legacy 5-level keys (`critical`, `info`) are still accepted on read for backward compatibility with older logs and normalized to `high`/`low`; new output is strictly 3-level.

**FixEvalDetail**: `{ path: string, line: int, findingId?: string, verdict: "not_attempted" | "attempted_failed" | "resolved" | "re_detected", reasoning?: string, durationMs: number, usage: UsageStats }`

### Output Location

- Always-on: `.warden/logs/{runId8}-{ISO-datetime}.jsonl` (repo-local, always written)
- Additional copy: `--output path/to/file.jsonl` (useful for CI artifact collection)
- Completion marker: `{logFile}.done` sidecar, written after the run finalizes

### Write Timing

The file is initialized (truncated to empty) before any skill runs, then each chunk record is appended via `fs.appendFileSync` from the runner's chunk-complete hook the moment that chunk finishes. No summary trailer is appended to new logs. At finalization, Warden rewrites the file from the final post-processed reports so durable JSONL matches deduplication, merge, and suggested-fix quality gates. If a skill produced no chunks, finalization writes a synthetic chunk record with `status: "skipped"` or `status: "error"` so the outcome is still durable. Each live record is emitted with one synchronous append call, so parallel callbacks in the Warden process cannot interleave partial JSON records.

Crash safety falls out for free: if the process dies mid-run, every chunk that completed is already on disk and `parseJsonlReports` will reconstruct reports from those chunks. The absence of the `.done` sidecar is the signal that the run is still in progress (or crashed); `warden runs follow` uses this to auto-resolve "the active session" when called without an argument.

Per-chunk `run.durationMs` is the elapsed-since-start at the moment that record was emitted. The chunk-level `durationMs` is only the chunk execution time. Readers that need run totals derive them from the maximum `run.durationMs` seen in the file (`parseJsonlReports` already does).

## 4. `--json` Output

`--json` outputs the JSONL log file contents to stdout. This is the same format as the always-on `.warden/logs/` file: one chunk result per line. See Section 3 for the full JSONL specification.

After the run finalizes, `--json` reads the on-disk log file back and writes it to stdout, preserving the per-chunk `run.durationMs` snapshots. If the file isn't readable (init failed, mid-run write errors leaving the file empty), `--json` falls back to rendering the same chunk-record shape in memory.

## 5. Formatting Functions

All reporters use shared formatters from `src/cli/output/formatters.ts`.

| Function | Input | Output | Example |
|----------|-------|--------|---------|
| `formatCost(costUSD)` | `number` | `$X.XX` (always 2 decimals) | `$0.09`, `$1.50`, `$0.00` |
| `formatTokens(tokens)` | `number` | Compact count | `850`, `1.2k`, `1.5M` |
| `formatDuration(ms)` | `number` | Human duration | `250ms`, `1.5s`, `12.3s` |
| `formatElapsed(ms)` | `number` | Elapsed prefix | `+250ms`, `+1.5s` |
| `formatLocation(path, start?, end?)` | `string, number?, number?` | File location | `src/a.ts:10`, `src/a.ts:10-20` |
| `formatUsage(usage)` | `UsageStats` | TTY usage | `2.5k in / 450 out . $0.09` |
| `formatUsagePlain(usage)` | `UsageStats` | Plain usage | `2.5k input, 450 output, $0.09` |
| `formatStatsCompact(ms?, usage?, aux?)` | Mixed | One-line stats | `T 1.5s . 2.5k in / 450 out . $0.09` |
| `formatAuxiliarySuffix(aux)` | `AuxiliaryUsageMap` | Aux breakdown | `(+extraction: $0.01, +dedup: $0.00)` |
| `formatFindingCounts(counts)` | `Record<Severity, number>` | TTY colored | `2 findings: . 1 high  . 1 medium` |
| `formatFindingCountsPlain(counts)` | `Record<Severity, number>` | Plain text | `2 findings (1 high, 1 medium)` |
| `formatSeverityBadge(severity)` | `Severity` | Colored dot + text | `. (high)` |
| `formatSeverityPlain(severity)` | `Severity` | Bracketed | `[high]` |
| `formatConfidenceBadge(confidence?)` | `Confidence \| undefined` | Colored bracketed (empty if undefined) | `[high confidence]` |
| `countBySeverity(findings)` | `Finding[]` | `Record<Severity, number>` | `{ high: 1, medium: 0, low: 0 }` |
| `pluralize(count, singular, plural?)` | `number, string` | Pluralized word | `file` / `files` |

## 8. Action Workflow Logging

The GitHub Action workflow (`pr-workflow.ts`) runs the same Plain reporter for skill execution, then performs additional workflow steps that produce their own logging. These are not reporter events. They use `logAction()` and `warnAction()` from `tty.ts`, which output in Plain format (`[ISO-8601] warden: {msg}`), plus GitHub-specific protocols.

### Context and Configuration

Inside `::group::` blocks with raw `console.log()`:

```
::group::Building event context
Event: pull_request
Workspace: /home/runner/work/repo/repo
::endgroup::

::group::Loading configuration
Config path: warden.toml
::endgroup::

::group::Matched triggers
- security-check: security-lint
- style-guide: style
::endgroup::
```

If no triggers matched: `No triggers matched for this event` and early return.

### Skill Execution

Each trigger runs inside a `::group::`:

```
::group::Running trigger: security-check (skill: security-lint)
[2026-02-08T14:30:45.123Z] warden: Running security-lint (3 file(s))...
[2026-02-08T14:30:46.500Z] warden:   security-lint > src/api/auth.ts done 1.8s $0.01 1 finding(s)
[2026-02-08T14:30:48.600Z] warden: security-lint completed in 4.2s - 2 findings (1 high, 1 medium)
Found 2 findings
::endgroup::
```

This is identical to Plain reporter output plus a bare `Found {N} findings` line from the trigger executor. The `::group::` wrapper is added by the trigger executor.

### Deduplication

```
[2026-02-08T14:30:49.000Z] warden: Found 12 existing comments for deduplication (8 Warden, 4 external)
[2026-02-08T14:30:49.100Z] warden: Found 3 duplicate findings for security-check
[2026-02-08T14:30:49.200Z] warden: Updated 2 existing Warden comments with skill attribution
[2026-02-08T14:30:49.300Z] warden: Added reactions to 1 existing external comments
```

On failures: `[...] warden: WARN: Failed to process 1 duplicate actions`
